Average QPS = DAU x requests / 86,400 seconds in a day. Peak factor lifts the average to a daytime or spike peak (2 to 3x for steady consumer, 5x or more for bursty). Read share splits the load into reads vs writes, which usually have very different cost and scaling paths.

Stored bytes per day = write QPS x object size x 86,400. Total storage = that x retention x replication factor. Replication of 3 is the common default for durability.

A cache absorbs the hits, so backend QPS = total QPS x (1 - hit rate). The 80/20 rule: holding the hot 20% of data in RAM commonly serves most reads. Servers = ceil(peak backend QPS / capacity per server), then add headroom.

Concurrency (Little's Law)

Little's Law: concurrent requests in flight = arrival rate x average latency. A pool of N slots sustains at most N / p99 requests per second. If demand exceeds that, queues build and latency climbs, which is why "just add threads" has a ceiling.

Source & assumptions

  • QPS uses 86,400 seconds per day; storage and bandwidth use 30.4 days per month.
  • Powers-of-ten unit ladders (KB, MB, GB, TB, PB and Kbps, Mbps, Gbps) keep the back-of-napkin honest.
  • Little's Law (concurrency = arrival x latency) is a steady-state identity; the p99-based ceiling is a planning rule of thumb, not a queueing-theory proof.
  • Latency ladder is the classic "numbers every engineer should know" order of magnitude, not measured on any specific machine.
