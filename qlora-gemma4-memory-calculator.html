Gemma QLoRA Memory Calculator

Pick a Gemma model and the memory-saving levers; see the estimated peak GPU memory for QLoRA fine-tuning and which GPUs fit. Covers Gemma 2, 3, 3n, and 4 sizes. The per-component formulas are calibrated against this repo's Gemma 4 E4B empirical mitigations doc (T4 OOM on static, L4 OOM on the bf16 cross-entropy logits, A100 fits with headroom) and scale to the other sizes by their architecture.

Inference / serving memory

This tool covers QLoRA training. Inference and serving are a different memory regime (no optimizer or gradients, but a KV cache that grows with context length and concurrency), and purpose-built calculators already cover it. Use one of these:

  • Google Gemma memory requirements: official per-model inference footprint (BF16 / SFP8 / Q4_0).
  • Hugging Face Model Memory Utility: official Accelerate estimator for load, inference, and training memory of any Hub model.
  • LLM Model VRAM Calculator: interactive KV-cache and context-length sizing across quantizations.

Source & assumptions

Formulas calibrated empirically against the Gemma 4 E4B OOM events in the OOM mitigations doc (T4 ~10 GB static, L4 ~22 GB at step 2, A100 ~17 GB at step 0). Naive textbook estimates predict ~13 GB, but real measurements show 17-22 GB. Every other Gemma size reuses these same per-component formulas with its own architecture numbers (layers, hidden, vocab, GQA heads), so treat the non-E4B totals as architecture-scaled estimates rather than separately measured. Why the gap:

  • NF4: 0.55 bytes/param (4-bit code + per-block fp16 scale)
  • Embeddings: vocab × hidden × 2 bytes (kept fp16, not 4-bit quantized). The Gemma 3n and Gemma 4 E-variants add Per-Layer Embedding tables on top (none on dense Gemma 2 / 3)
  • LoRA: r × (in+out) per projection × layers, fp16
  • adamw_8bit: ~6 bytes/trainable param (m+v 8-bit + fp32 master)
  • paged_adamw_8bit: ~4 bytes/trainable param (master paged to CPU)
  • Activations: bs × seq × (hidden + mlp_intermediate) × layers × 6, ÷ 4 under Unsloth checkpointing. Unsloth's checkpointing keeps attn-block intermediates, so the reduction is ~4× not 8×. Per-layer holds residuals, layer norms, Q/K/V, attention scores, MLP gate/up/down intermediates.
  • bnb dequant scratch (accumulated): per-tile size × 3. Multiple layers' dequant'd fp16 tiles coexist in PyTorch's caching allocator. Sequence-independent, so dropping seq does NOT help this.
  • Cross-entropy logits + backward: bs × seq × vocab × 2 × 3. The large 256k-262k Gemma vocab makes this dominant on L4. The ×3 covers forward logits + grad-w.r.t.-logits + log_softmax intermediates. This is the line that pushes L4 over 22 GB even with sort + checkpointing on.
  • cuBLAS + autocast workspaces: ~2.5 GB. The LM head matmul over the 256k-262k vocab needs larger cuBLAS scratch than typical hidden-to-hidden ops; bf16 autocast also keeps brief fp32 intermediates for stability (softmax, layer norm).
  • Framework overhead: 1.5 GB with expandable_segments, 2.5 GB without, from allocator fragmentation, PEFT/Unsloth wrapper allocations, autograd graph storage.
  • KV cache: not part of training. A fine-tuning step is a single teacher-forced forward pass, so there is no autoregressive KV cache; the K/V tensors are already inside the activations line. KV cache only matters for inference, which the linked calculators in the "Inference / serving memory" panel cover.
