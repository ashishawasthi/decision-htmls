Cloud Audit Logs (Data Access) in agentic systems

Cloud Audit Logs is the platform-level record of who did what, where, and when across Google Cloud. Three of its four streams are always on and free; the Data Access stream, which records reads and writes of your data in managed services, is the one you opt into and pay for. The design rule that follows: turn Data Access logging on for the managed stores that hold sensitive data once the audience is regulated or external, and keep it separate from the application audit log, which records what the agent itself did.

Not the same as the immutable application log. The Cloud Logging (WORM) control records the agent's own prompts, tool invocations, and decisions, written by your application. Cloud Audit Logs records who touched the infrastructure and the managed data stores: IAM changes, and reads and writes against AlloyDB, Spanner, BigQuery, Cloud Storage, Secret Manager, and Cloud KMS keys. The two are complementary, not substitutes. Reflects Google Cloud guidance as of June 2026; verify against current docs.

The four streams

StreamDefaultCaptures
Admin ActivityAlways on, free, cannot disableControl-plane changes: IAM grants, resource and config changes.
System EventAlways on, freeGoogle-initiated changes (for example autoscaling).
Policy DeniedOn by default, freeAccesses blocked by org policy, VPC Service Controls, or context-aware access.
Data AccessOff by default (except BigQuery), billed, opt-inReads and writes of data in managed services, plus KMS key use and Secret Manager access.

Because the first three streams cost nothing and need no setup, the only real design decision is the Data Access stream: which services to enable it on, and how to keep its volume in check.

When to use, when not

Good fit

  • Regulated or external audiences where you must prove who read or changed sensitive data in managed stores.
  • Compliance regimes (SOC 2, HIPAA, PCI, public sector) that require data-access accountability, not just control-plane change history.
  • Forensics: reconstructing exactly which records a run or service account accessed, and when.
  • Detecting misuse of a runtime service account, such as unexpected reads of a store it should not touch.

Avoid or take care

  • As a replacement for the application audit log: it records platform access, not the agent's prompts, tools, or decisions. Keep both.
  • Enabling Data Access logs project-wide at full verbosity in a high-volume design: the billed volume can balloon. Scope to the stores that hold sensitive data.
  • As a low-latency alerting feed: it is an audit trail, not a metrics pipeline. Use Cloud Monitoring for that.
  • Budgeting for Admin Activity coverage: it is already on and free, so it is not a provisioned line.

Choosing Cloud Audit Logs vs. alternatives

What you needUse instead
Who read or wrote managed data (control plane plus data plane)Cloud Audit Logs with Data Access enabled on those services
The agent's own prompts, tool calls, and decisions, kept tamper-evidentCloud Logging with a locked WORM bucket
Latency, spans, and errors across the request pathCloud Trace with Cloud Monitoring
Cross-source threat correlation and detectionsExport to Chronicle or a third-party SIEM
IAM changes and resource config history onlyAdmin Activity logs (already on, free)

Integration patterns and tuning

  • Enable Data Access logs only on the services that hold sensitive data (the managed state store, BigQuery, Cloud Storage, Secret Manager, KMS), not project-wide, to keep the billed volume legible.
  • Pair it with the other tier-gated controls: VPC Service Controls draws the perimeter, CMEK encrypts the managed stores, and Data Access logs record who reached them. Perimeter, encryption, and access accountability are the three legs of the managed-store story.
  • Route audit logs to a dedicated, access-restricted bucket or sink, and turn on Data Access logging for that bucket too, so reads of the audit trail are themselves recorded.
  • Redact before ingest where log bodies can carry PII (resource names, query parameters) with Cloud DLP, since a locked audit sink cannot delete a leaked value.
  • Trim high-noise, low-value entries with exclusion filters, and sink long-term retention to lower-cost storage for the required number of years.
  • Keep the runtime service account least-privileged so the trail stays readable: fewer principals make the reads easier to attribute.

How it maps to this tool

The designer turns Data Access audit logging on automatically when the audience and sensitivity tier is regulated or higher, or when the system is external-facing, the same trigger that drives the immutable application audit log it complements. It then adds a Cloud Audit Logs (Data Access) line to the bill of materials and draws a data-access edge from each managed store to a Cloud Audit Logs node in the architecture diagram, alongside the CMEK encryption edges on the same stores. Below the regulated tier (internal, low-stakes) it stays off, since the always-on Admin Activity, System Event, and Policy Denied streams already cover control-plane history at no cost. Only the billed Data Access stream is modeled in the cost estimate.
