Document parsing and extraction

Document AI in agentic systems

Document AI runs OCR plus layout and entity extraction that turns PDFs, scans, and forms into structured, chunkable text: it recovers reading order, tables, and form fields that a plain text dump loses. The design rule that follows: use it at ingestion time to prepare a clean document corpus for embedding, not on the live request path, and treat its per-page cost as a real line item in the ingestion budget. Reflects Google Cloud guidance as of May 2026; verify against current Google Cloud docs.

Role

Document AI sits at the front of the ingestion pipeline as the parsing stage. It converts a raw document corpus into structured text and layout that downstream chunking and embedding can use. The role does not flip between an interactive assistant and task automation: in both cases it is an offline, batch-oriented preprocessor, and what changes is only how often you reindex.

When to use, when not

Good fit

  • A corpus of PDFs, scans, or photographed documents where text is not already machine-readable and OCR is required.
  • Documents whose meaning depends on layout: tables, multi-column pages, forms, and key-value pairs that a naive extractor would scramble.
  • Preparing a corpus for embedding and RAG, where clean reading order and preserved tables directly improve chunk quality and retrieval.
  • Structured extraction of named entities (dates, amounts, parties) from forms and contracts for downstream grounding.

Avoid

  • On a sub-second request path: parsing is an ingestion step, not a per-query operation.
  • Corpora that are already clean text (Markdown, HTML, well-structured exports). A text-extraction library is cheaper and adequate.
  • Very high document volumes where per-page parsing cost is not budgeted: it can dominate ingestion spend.
  • When the platform's built-in parsing already meets quality needs (for example Vertex AI Search ingesting PDFs directly), adding a separate parser may be redundant.

Choosing Document AI vs. alternatives

What you need Use instead Typical latency or cost
Plain text from already-digital files, layout not critical Text-extraction libraries (for example pdfplumber, Apache Tika) near-free compute, but loses tables and layout
End-to-end managed RAG with adequate built-in parsing Vertex AI Search built-in document parsing folded into the search index cost
High-fidelity OCR plus layout, tables, and entities Document AI per page processed (a few cents per page, tier-dependent)
Existing third-party OCR contract or on-prem requirement Third-party OCR engine vendor-dependent; verify layout and table support

Latency and cost profile

  • Latency: seconds per document for online processing, longer for full batch jobs. This is fine because parsing runs at ingestion, off the request path, so nothing waits on a sub-second response.
  • Cost: billed per page processed, with the rate depending on the processor type (for example general OCR versus specialized form or layout parsers). Roughly a few cents per page as of May 2026; verify against current Google Cloud pricing. At high document volumes this per-page charge can dominate total ingestion cost, so size it against the corpus before committing.

Integration patterns and tuning

  • Parse then chunk then embed: run Document AI first to recover clean text and layout, then chunk on structural boundaries (sections, tables, paragraphs) rather than fixed character windows, then embed. Better parsing yields better chunks and better retrieval.
  • Batch over online: for an initial corpus load or a reindex, use batch processing across the whole corpus rather than per-document online calls. It is cheaper and matches the offline nature of ingestion.
  • Pick the right processor: use the general or layout parser for broad corpora and a specialized processor only where structured entity extraction earns its higher per-page cost.
  • Control volume: deduplicate and filter the corpus before parsing so you do not pay per page for documents you will never retrieve.

How it maps to this tool

In the designer, Document AI is the parsing stage inside the Ingestion pipeline, which is drillable, and it backs the Doc corpus data source. When the corpus is PDFs or scans, the pipeline routes them through Document AI before chunking and embedding.

It is an ingestion cost, not a serving cost. Because parsing is per page and runs offline, the tool accounts for Document AI in the ingestion budget rather than on the request path, so a chat-heavy assistant does not get charged parsing latency per query, while a large-corpus build shows the per-page cost where it actually lands.
