On-prem system of record

On-prem DB in agentic systems

An on-prem DB is a legacy or on-premises database reached over private connectivity (Cloud Interconnect, Cloud VPN, or Private Service Connect), used when the system of record cannot move to the cloud. The one fact that drives the design: every read crosses your private link, so latency is variable and often higher than a cloud-native store. The rule that follows: in an interactive request path, prove it fits the latency budget; in asynchronous automation it is usually fine. Reflects Google Cloud guidance as of May 2026.

Role by purpose

The right place for an on-prem DB flips with the top-level purpose you pick in the tool, because the constraint is the network round-trip rather than the data itself.

Purpose Recommended role Fit
Interactive assistant Grounding source only if it fits the latency budget, otherwise replicate or cache hot data into a cloud store and read that on the hot path Conditional
Task automation Primary system of record; asynchronous runs absorb the connectivity latency Strong

When to use, when not

Good fit

  • The authoritative record lives on-prem and residency, licensing, or security policy prevents it from moving to the cloud.
  • Asynchronous automation with minute-scale budgets, where a slower or occasionally variable read does not break an SLO.
  • Reads that comfortably fit the latency budget once you have measured the round-trip across Interconnect or VPN, not guessed it.
  • A bridge during migration: the agent reads the on-prem source today while hot data is replicated into a cloud store for the eventual cutover.

Avoid

  • A sub-second interactive hot path that depends on every request reaching the on-prem DB. A slow link or a blip will blow the budget.
  • Designs with no fallback when connectivity degrades. A single private link is a reliability and availability risk on the request path.
  • High-fan-out per-request lookups, where each call pays the round-trip and they stack up.
  • New greenfield data that has no residency constraint. Put it in a cloud-native store from the start.

Choosing On-prem DB vs. alternatives

What you need Use instead Typical latency / cost
Hot data off the request path, low ms reads Replicate or cache into AlloyDB, Spanner, or Bigtable single-digit ms
Authoritative record that cannot move On-prem DB over Interconnect, VPN, or Private Service Connect variable, often tens of ms or more plus link cost
Fresh changes streamed out of the legacy system Change data capture into Pub/Sub, then a cloud store near-real-time, pipeline cost
Lowest, most stable round-trip to on-prem Dedicated or Partner Interconnect over Cloud VPN committed bandwidth, higher fixed cost

Latency and cost profile

  • Latency: variable and often higher than a cloud-native store, set by the private link and the distance to your data center. Cloud VPN runs over the public internet and varies more; Dedicated Interconnect gives a more predictable round-trip. Treat the figure as something to measure, not assume.
  • Cost: dominated by the connectivity itself, not per-query. Interconnect carries port and VLAN attachment charges plus egress; Cloud VPN carries tunnel and egress charges. Rough ranges only as of May 2026, and they scale with committed bandwidth, so verify against current Google Cloud docs.

Integration patterns and tuning

  • Keep it off the hot path: for an assistant, replicate or cache the hot, frequently read data into a cloud store (AlloyDB, Spanner, or Bigtable) and read that synchronously. Reach the on-prem DB only when you must and the budget allows.
  • Stream, do not poll: use change data capture into Pub/Sub to push fresh changes out of the legacy system rather than having the agent reach back in on every request.
  • Harden the link: prefer Dedicated or Partner Interconnect for a predictable round-trip, and design for connectivity loss with a fallback or a clear degraded mode rather than a hard dependency.
  • Private and contained: reach the source over Private Service Connect or private connectivity, set tight timeouts and retries, and keep the access path inside your VPC.

How it maps to this tool

The On-prem DB data source pairs with the on-prem deployment option in the designer. Selecting it tells the tool the system of record sits behind private connectivity rather than in a managed cloud store.

Latency is a decision, not a default. On the assistant side the tool treats an on-prem read as conditional: confirm it fits the request budget, or move hot data into a cloud store and read that instead. On the automation side it is a strong fit, since asynchronous runs absorb the variable connectivity latency.
