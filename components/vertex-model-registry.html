Model and version governance

Model Registry in agentic systems

Model Registry is the system of record for what is in production: it registers, versions, and stages models (and, alongside it, prompt versions) with lineage, approvals, and rollback. Under a model-risk regime the question "exactly which model and prompt served this decision, and who approved it" must have a precise, auditable answer, and an ungoverned config string cannot give one. The design rule that follows: when you are subject to model-risk governance, pin every production model and prompt to a registered, approved version so promotion, audit, and rollback all run through one controlled path. Reflects Google Cloud guidance as of May 2026; verify against current Google Cloud docs.

Role

The role does not change with the top-level purpose. For both an interactive assistant and a task automation system, Model Registry is the governed catalog and promotion control point for models and versions. Purpose may change how often you ship a new version, but not the need to register, approve, and be able to roll it back.

When to use, when not

Good fit

  • Model-risk governance regimes (regulated finance, healthcare, public sector) that require you to pin, approve, and audit exactly what is in production.
  • Champion-challenger lifecycle management: track versions, stage a challenger, and promote or roll back with a clear trail.
  • Lineage and reproducibility: tying a production version back to its training data, evaluation, and the person who approved it.
  • Fast, controlled rollback to a known-good version when a release regresses.

Avoid

  • Throwaway prototypes and demos where formal versioning and approvals add overhead without payoff.
  • As a serving or inference engine: the registry governs versions; Agent Platform endpoints serve them.
  • As your evaluation system: it stores results and approvals but does not compute quality. Use Agent Platform Evals.
  • As the audit log of runtime events: that is Cloud Logging. The registry governs artifacts, not request traffic.

Choosing Model Registry vs. alternatives

Measuring quality of a version before promotion Agent Platform Evals (feeds approval decisions)

Integration patterns and tuning

  • Register every production candidate: push each model and prompt version into the registry with metadata, rather than referencing a model by a loose string in app config.
  • Gate promotion on eval and approval: require a passing Agent Platform Evals run and a recorded human approval before a version is promoted to the production alias.
  • Serve through an alias, not a hard-coded ID: point the endpoint at a stable production alias so a rollback is repointing the alias to the previous version, fast and auditable.
  • Track lineage end to end: link each version to its dataset, evaluation results, and approver so an auditor can reconstruct exactly what served a decision and why.
  • Version prompts too: treat the prompt as a governed artifact alongside the model, since in an agentic system the prompt changes behavior as much as a weight update does.

How it maps to this tool

This component backs the Model risk governance override. When the scenario calls for model-risk controls, the rules engine turns this on so production models and prompts are pinned to registered, approved versions instead of floating on whatever config happens to be deployed.

It is the governance counterpart to eval. The designer treats Agent Platform Evals as the measure and the registry as the gate: eval scores a version, the registry records the approval and pins what ships, and together they give you a defensible, reversible promotion path.
