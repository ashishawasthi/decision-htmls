Immutable audit logging

Cloud Logging (WORM) in agentic systems

Cloud Logging centralizes logs across the agent stack, and a locked retention bucket turns a log bucket into write-once-read-many storage: once a retention period is locked, no one, not even a project owner, can shorten it or delete entries before it expires. That single property is what makes it admissible as a tamper-evident audit trail. The design rule that follows: reach for it when an auditor or regulator must trust that the record was not edited after the fact, and pair it with redaction so the immutable record never freezes raw PII in place. Reflects Google Cloud guidance as of May 2026; verify against current Google Cloud docs.

Role

The role does not change with the top-level purpose: in both an interactive assistant and a task automation system, Cloud Logging with a locked bucket is the tamper-evident system of record for audit and forensic events. What changes is volume and event mix, not the role.

When to use, when not

Good fit

  • Regulated audit requirements (finance, healthcare, public sector) where you must prove the log was not altered or pruned after an event.
  • Security forensics and incident response, where a defensible timeline of agent actions, tool calls, and data access matters.
  • Recording prompt, tool-invocation, and decision events for governance regimes that demand retention for a fixed number of years.
  • Meeting legal hold or chain-of-custody expectations on operational logs.

Avoid

  • General app debugging where standard, non-locked buckets are cheaper and stay flexible.
  • Storing raw PII or secrets in the log body: a locked bucket means you cannot delete the bad entry. Redact first.
  • High-cardinality metrics or traces: use Cloud Monitoring and Cloud Trace, not the audit log.
  • As a queryable feature store or analytics warehouse. Route to BigQuery for analysis instead.

Alternatives, and how to choose

What you need Use instead Typical latency / cost
Tamper-evident, retention-locked audit trail Cloud Logging with a locked retention bucket per GB ingested plus storage
Operational logs you may need to edit or expire early Standard Cloud Logging bucket (no lock) per GB, lower retained volume
Central SIEM correlation across many sources Export sink to Chronicle, Splunk, or a third-party SIEM export is low cost; SIEM licensing dominates
Long-term cold archive at lowest storage cost Sink to a Cloud Storage bucket with object retention / Bucket Lock cents per GB-month, archival tier

Latency and cost profile

  • Latency: not on any request path. Logging is asynchronous and fire-and-forget from the agent's perspective, so it does not add to user-facing latency. Treat write durability, not speed, as the requirement.
  • Cost: driven by GB ingested and GB stored over the retention window. A common pattern is a generous free ingestion allowance per month, then a per-GB ingestion charge (roughly single-digit dollars per GB as of May 2026) plus per-GB-month storage beyond the included period. Locked long retention is the main cost lever, so log deliberately and redact aggressively. Verify current rates against Google Cloud pricing.

Integration patterns and tuning

  • Lock the retention, on purpose: create a dedicated log bucket, set the retention period, then lock it. Locking is irreversible, so size the period to the longest applicable regulatory requirement before you commit.
  • Redact before it lands: run log payloads through Cloud DLP (Sensitive Data Protection) de-identification so PII is masked or tokenized before the immutable write. A locked bucket cannot delete a leaked secret, so the guardrail has to sit upstream.
  • Separate audit from debug: route audit and security events to the locked bucket with a log sink and filter, and keep noisy application logs in a standard, shorter-retention bucket to control cost.
  • Tighten access: restrict read access to the audit bucket with IAM, and enable Data Access audit logs so reads of the audit trail are themselves recorded.
  • Fan out where needed: add an export sink to a SIEM or to BigQuery for correlation and analysis, while the locked bucket remains the authoritative untouched copy.

How it maps to this tool

This component backs the Immutable audit log override. When you raise data sensitivity to the regulated level, the rules engine locks this control on rather than leaving it optional, because at that sensitivity a tamper-evident trail is a hard requirement, not a nice-to-have.

It travels with redaction. The designer pairs the immutable audit log with PII redaction (Cloud DLP) under regulated sensitivity, so the record you cannot edit is also a record that never captured raw PII in the first place.
