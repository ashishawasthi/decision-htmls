Dataflow in agentic systems

Dataflow is serverless Apache Beam for streaming and batch data processing: one programming model that autoscales workers and runs the same pipeline over a live stream or a finite dataset. The design rule that follows: make it the ETL backbone of a separated ingestion pipeline (parse, chunk, embed, index) so that backfills and full reindexing run as their own jobs and never disturb live serving traffic. Reflects Google Cloud guidance as of May 2026; verify against current Google Cloud docs.

Role

Dataflow is the data-processing engine that moves a corpus through parse, chunk, embed, and index. The role is the same whether the system is an interactive assistant or task automation: it is an offline, scale-out ETL layer that sits behind serving. What it buys you is separation: ingestion and reindexing become jobs you can run, scale, and retry without touching the request path.

When to use, when not

Good fit

  • A large corpus where ingestion is separated from serving and a reindex must run without slowing or risking live traffic.
  • Backfills and full reindexing: reprocess the whole corpus, recompute embeddings after a model change, and rebuild the index as a discrete job.
  • Streaming ingestion: documents or events arriving continuously that need parse, chunk, and embed in near real time.
  • Pipelines that need autoscaling and exactly-once processing over millions of items without you managing worker fleets.

Avoid

  • Small or one-off ingestion where a single Cloud Run job or script is simpler and cheaper to operate.
  • Pure step orchestration (run A, then B, then C) with little data crunching: a workflow or scheduler fits better than a Beam pipeline.
  • On the synchronous request path: Dataflow is batch and streaming ETL, not a per-query service.
  • Teams with no Apache Beam familiarity and a tiny corpus, where the learning curve outweighs the benefit.

Choosing Dataflow vs. alternatives

What you need Use instead Typical latency or cost
Simple, smaller-scale ingestion with no streaming Cloud Run jobs per vCPU and memory while running; lower floor
Orchestrate multi-step pipelines and dependencies Cloud Composer (Airflow) or Workflows environment or per-step cost; pairs with, not replaces, Dataflow
Large-scale streaming or batch ETL that autoscales Dataflow per vCPU-hour plus memory and shuffle
Quick experiment or tiny corpus Manual scripts near-free, but no autoscaling, retries, or operability

Latency and cost profile

  • Latency: minutes to hours for a batch reindex, seconds of processing lag for streaming. It runs off the request path, so this latency sits in ingestion, not in any serving SLO.
  • Cost: billed per vCPU-hour plus memory, local disk, and data shuffled, summed across autoscaled workers. Spend tracks how much data you process and how parallel you run, so a full reindex is a visible, bursty cost rather than a steady one. Verify current rates against Google Cloud pricing.

Integration patterns and tuning

  • Separated ingestion pipeline: Dataflow reads the corpus, parses (often via Document AI), chunks, calls the embedding model, and writes vectors to the index, all as a job decoupled from serving.
  • Reindex without downtime: build a fresh index in a new job, then swap serving over to it, so live traffic never reads a half-built index.
  • Batch versus streaming: use batch for backfills and periodic full rebuilds; use streaming when documents arrive continuously and freshness matters.
  • Tune throughput and cost: right-size machine types, cap maximum workers, and use templates for repeatable jobs. Pair with Cloud Composer or Workflows when ingestion is several ordered stages.

How it maps to this tool

Dataflow is what the designer reaches for behind the Ingestion separation override: the choice to run parse, chunk, embed, and index as their own pipeline rather than inline with serving.

The rules engine locks it on for a large corpus. Once corpus size crosses the threshold where inline ingestion would threaten the serving path, the tool turns Ingestion separation on and keeps it on, so a heavy reindex runs as a Dataflow job instead of competing with live traffic.
