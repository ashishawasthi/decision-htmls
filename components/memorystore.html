Managed Redis / Memcached cache

Memorystore in agentic systems

Memorystore is a managed in-memory cache (Redis and Memcached) with sub-millisecond access. The one fact that drives the decision: it is fast because it is in memory, which also means it is a cache, not a durable system of record. The design rule that follows: put it in front of slower stores for response and semantic caching, session and short-term agent state, and rate-limit counters; keep durable, must-not-lose data on a persistent store behind it. Reflects Google Cloud guidance as of May 2026; verify against current Google Cloud docs.

Role

The role is the same across the tool's top-level purposes: Memorystore is the low-latency cache and short-term state layer. It absorbs repeated reads (response and semantic caches), holds session and short-term agent state, and serves fast counters for rate limiting, sitting in front of the durable stores rather than replacing them.

When to use, when not

Good fit

  • Response cache and semantic cache to short-circuit repeated or near-duplicate model calls and cut latency and cost.
  • Session and short-term agent state that needs the fastest possible access and can tolerate eviction or a TTL.
  • Rate-limit counters and other hot, frequently updated values that benefit from in-memory atomic operations.
  • A read-through or look-aside cache in front of a slower database, so the hot path skips the heavier store.

Avoid

  • Durable per-request point lookups that must not be lost (profiles, entitlements, feature data at scale). Use Bigtable for single-digit ms lookups with durability.
  • A transactional system of record or anything needing strong durability guarantees. Use Spanner, AlloyDB, or Cloud SQL.
  • Large analytical scans. Use BigQuery, off the request path.
  • Serving static assets to end users. A CDN is the right layer for that.

Choosing Memorystore vs. alternatives

What you need Use instead Typical latency / cost
Durable per-request key lookups at massive scale Bigtable single-digit ms; per node and storage
Caching tightly coupled to query results In-database caching (buffer cache, materialized views) low ms; part of the database cost
Caching static responses near end users A CDN (Cloud CDN) tens of ms at the edge; per egress and requests
Sub-ms cache, session and short-term state, counters Memorystore sub-millisecond in memory; per instance capacity

Latency and cost profile

  • Latency: sub-millisecond for in-memory reads and writes within the region, which is why it is the layer you reach for when even single-digit milliseconds is too slow. Network round-trip from the client adds to that, so co-locate the cache with the workload.
  • Cost: billed per provisioned in-memory capacity (instance size and tier), so you pay for the memory footprint and availability tier rather than per request. Because it is in memory, capacity is the main cost lever and the reason caches use TTLs and eviction. Treat any figure as a rough range and verify against current Google Cloud pricing.

Integration patterns and tuning

  • Look-aside response cache: check the cache first, call the model or database on a miss, then write the result back with a TTL. This is the simplest big win for repeated queries.
  • Semantic cache: key on an embedding or a normalized form of the prompt so near-duplicate queries hit the cache, returning a stored answer instead of a fresh model call.
  • Session and short-term state: store conversation and working state with a TTL so stale sessions expire on their own; keep anything that must survive on a durable store.
  • Rate limiting: use atomic counters with expiry windows for per-user or per-tenant limits, kept in memory for speed.
  • Eviction policy and sizing: pick an eviction policy that matches the access pattern and size the instance so the working set fits in memory; pick the high-availability tier when an outage would hurt.

How it maps to this tool

In the designer this is the state store Redis option and the response-caching layer. Selecting it models a sub-millisecond cache and short-term state store in front of the durable stores, plus the cache that short-circuits repeated model calls on the hot path.

Durable point lookups belong on Bigtable. Memorystore is a cache, so per-request key lookups that must be durable should land on Bigtable (single-digit ms, durable, at massive scale), not on an in-memory cache and not on an analytical warehouse. This complements the guidance on the BigQuery page that point lookups do not belong in BigQuery.
