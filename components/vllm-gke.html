Self-hosted open-weight serving

vLLM on GKE in agentic systems

vLLM serves open-weight models on GPU nodes with paged attention and continuous batching, which pack many concurrent requests onto each accelerator and raise tokens-per-second-per-GPU; on Google Kubernetes Engine you own the cluster, the GPUs, and the operations. The design rule that follows: self-host only when the driver is real: data residency, control over an open model, or cost at sustained high volume. Below high steady utilization, a managed token API usually wins. Reflects Google Cloud guidance as of May 2026; verify against current Google Cloud docs.

Role

vLLM on GKE is the self-hosted serving tier for the model itself. It is the path you take when a managed model API does not fit a hard constraint. The role does not change between an interactive assistant and task automation: in both it is the inference backend. What changes is how you size the GPU fleet, since an interactive workload is latency-bound while batch automation can run hotter for throughput.

When to use, when not

Good fit

  • Data residency or isolation rules that require the model to run inside your own VPC and cluster.
  • Open-weight model control: a specific checkpoint, a fine-tune, or a version you must pin and not have changed under you.
  • Sustained high volume where steady GPU utilization makes node-hours cheaper than per-token API pricing.
  • Teams with the platform skills to run GKE, GPUs, autoscaling, and upgrades in production.

Avoid

  • Spiky or low-to-moderate traffic: idle GPUs are expensive, and a managed pay-per-token API is usually cheaper and simpler.
  • Teams without GPU and Kubernetes operations capacity: the undifferentiated work is real.
  • When a managed frontier model is required for quality and no suitable open-weight model exists.
  • Fast prototypes and pilots, where standing up a GPU fleet is premature.

Latency and cost profile

  • Latency: driven by model size, accelerator, quantization, and batch fullness. Continuous batching keeps throughput high under load but, pushed too hard, it trades per-request latency for tokens-per-second, so size for peak decode throughput at your target tail latency.
  • Cost: GPU node-hours, paid whether or not the GPUs are busy. The fleet is sized to peak decode throughput, so the economics turn on utilization: high and steady favors self-hosting, spiky or low favors a managed token API. Verify current accelerator rates against Google Cloud pricing.

Integration patterns and tuning

  • Throughput features: vLLM's paged attention and continuous batching are what make GPU self-hosting economical; without them you would need far more accelerators for the same load.
  • Quantization: serving a quantized checkpoint cuts memory and can raise throughput per GPU, often the single biggest lever on fleet size and cost, at some accuracy cost to validate.
  • Fleet sizing: choose the accelerator, set quantization, pick a pricing tier (on-demand, committed use, or Spot), and target a utilization that holds tail latency, then size node count to peak decode throughput.
  • Autoscaling and headroom: scale GPU node pools on load and keep enough headroom that batches do not overflow at peak, which is when latency degrades fastest.

How it maps to this tool

Selecting the self-host deployment path in the designer routes the model serving tier to vLLM on GKE and opens GPU fleet sizing: accelerator type, quantization, pricing tier, and target utilization.

The sizing inputs drive the cost. The tool sizes the fleet to peak decode throughput from those inputs, so changing accelerator, quantization, pricing tier, or utilization moves the estimate. That is what lets you compare self-hosting against a managed token API at your own volume instead of in the abstract.
