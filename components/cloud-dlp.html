Sensitive data protection (PII)

Cloud DLP in agentic systems

Cloud DLP (Sensitive Data Protection) discovers, classifies, and de-identifies sensitive data: redact, mask, tokenize, or apply format-preserving encryption. The design rule that follows: redact PII in the gateway and at ingestion, so the model never sees raw sensitive data and none of it lands in logs. Reflects Google Cloud guidance as of May 2026.

Role

The role does not meaningfully change with purpose: Cloud DLP is the de-identification step that removes or tokenizes PII before content reaches the model and before it is written to logs or storage. For an interactive assistant it sits inline in the request and response path; for task automation it runs at ingestion and on outputs. Either way the job is the same, and either way it is complementary to safety screening, not a substitute for it.

When to use, when not

Good fit

  • Any regulated or strict-PII workload where prompts, retrieved content, or outputs may carry personal data that must not reach the model in the clear.
  • Redacting before logging and tracing, so sensitive data is not persisted in observability or analytics stores.
  • Ingestion pipelines that scan and de-identify documents before they enter a corpus or vector store.
  • Reversible tokenization or format-preserving encryption when a downstream step must re-identify data under control.

Avoid

  • Fully synthetic or already-anonymized data with no personal information to protect.
  • Treating it as a safety filter: it removes sensitive data but does not stop prompt injection, jailbreaks, or malicious URLs. That is Model Armor.
  • Treating it as a network boundary: de-identification is not a perimeter. Data residency and exfiltration control come from VPC Service Controls.
  • Hard real-time paths with no budget for an inline inspection call, unless you scope the scan tightly or run it asynchronously.

Choosing Cloud DLP vs. alternatives

What you need Use instead Typical latency or cost
Context-aware PII discovery and de-identification across many infoTypes Cloud DLP inline inspection call; priced per unit processed
Cheap, narrow pattern matches on a few known formats DIY regex near-zero cost; brittle, misses context and edge cases
Basic built-in redaction bundled with the model Provider-native redaction bundled; fewer types, less control, less portable
Stop unsafe content (injection, jailbreaks, leakage) Model Armor (complementary) safety screening, not de-identification
Keep data inside a residency or exfiltration boundary VPC Service Controls (complementary) network perimeter, not field-level redaction

Latency and cost profile

  • Latency: an inline inspection and de-identification call on the content, so it adds a hop to the request path. Cost scales with how much you scan, so narrow the infoTypes and payload on interactive paths and push bulk scanning to ingestion.
  • Cost: priced per unit processed (by bytes or records inspected), so spend tracks data volume and how often you scan. It is usage-dependent; confirm the current per-unit rate against Google Cloud pricing.

Integration patterns and tuning

  • Redact at the gateway: de-identify prompts on the way in and responses on the way out at the Apigee or orchestrator layer, so raw PII never reaches the model and never enters request logs.
  • De-identify at ingestion: scan and redact documents before they enter a corpus or vector store, so the retrieval path cannot surface sensitive data later.
  • Pair with Model Armor: run both at the same seam. Guardrails stop unsafe content, DLP removes sensitive data; together they cover safety and privacy on the same hop.
  • Tuning: select the infoTypes that matter for your jurisdiction, choose a transform (redact, mask, tokenize, or format-preserving encryption) per re-identification need, and scope scans to the smallest payload to hold down latency and cost.

How it maps to this tool

The designer applies always-on DLP redaction at the gateway and turns on PII handling whenever sensitivity is set to regulated or strict-PII. Enabling it places de-identification in front of the model and ahead of logging.

It is a distinct control from its neighbors. The tool separates three jobs that are easy to conflate: Cloud DLP removes sensitive data, Model Armor (Safety guardrails) screens unsafe content, and VPC Service Controls draws the network perimeter. A strict-PII design typically switches on all three, since each closes a gap the others do not.
