Entity and relationship store

Knowledge graph in agentic systems

A knowledge graph is a store for entities and the relationships between them (Spanner Graph, or a third party such as Neo4j), enabling multi-hop reasoning, entity resolution, and lineage. The one fact that drives the design: it answers "how are these connected", which semantic vector search cannot. The rule that follows: reach for it when relationships and structured reasoning matter, and pair it with vector retrieval rather than treating one as a replacement for the other. Reflects Google Cloud guidance as of May 2026.

Role

The role holds across purposes: in both an interactive assistant and task automation, a knowledge graph is a structured grounding source for relationship and multi-hop questions. The difference is only depth of traversal: an assistant keeps hops shallow to stay in a sub-second budget, while automation can afford deeper walks. A single, well-tuned traversal is fast enough for either, which is why this stays one role rather than a per-purpose split.

When to use, when not

Good fit

  • Multi-hop reasoning: questions that walk several relationships (this customer, their accounts, the linked parties, the shared addresses).
  • Entity resolution: stitching records that refer to the same real-world thing across systems.
  • Lineage and impact analysis: tracing what depends on what, upstream and downstream.
  • Structured grounding that complements semantic retrieval: the graph supplies the connections, vector search supplies the unstructured context.

Avoid

  • Pure semantic similarity over text, where there is no relationship to traverse. Use vector RAG.
  • Simple, shallow joins over tabular data that relational SQL already handles cleanly.
  • Deep, unbounded traversals on a sub-second hot path. A wide multi-hop walk can blow the budget; bound the depth.
  • Standing up a graph for its own sake when no question actually needs relationship traversal.

Choosing Knowledge graph vs. alternatives

What you need Use instead Typical latency / cost
Multi-hop relationships, entity resolution, lineage Knowledge graph (Spanner Graph or Neo4j) ms to tens of ms per traversal
Semantic similarity over unstructured text Vector RAG (Vertex AI Vector Search) around 12 ms per query
Shallow joins and aggregates over structured rows Relational SQL (Spanner, AlloyDB, or Cloud SQL) low ms for indexed joins
Both connections and unstructured context Graph for traversal plus vector search for retrieval combined, run in parallel

Latency and cost profile

  • Latency: roughly milliseconds to tens of milliseconds per traversal for a bounded walk, which can fit an interactive budget. Latency grows with hop depth and fan-out, so an unbounded deep traversal is the thing to watch.
  • Cost: depends on the engine. Spanner Graph rides Spanner compute and storage pricing; a third-party store such as Neo4j carries its own licensing or instance cost. Rough ranges only as of May 2026; verify against current Google Cloud or vendor docs.

Integration patterns and tuning

  • Graph plus vector, in parallel: the common pattern is to run a graph traversal and a vector retrieval together, then feed both into the prompt. The graph contributes structure and the vector store contributes unstructured context.
  • Bound the traversal: cap hop depth and result fan-out on an interactive path so a wide walk cannot blow the latency budget. Reserve deeper walks for asynchronous automation.
  • Model for the questions: design nodes and edges around the relationships the agent actually needs to traverse, and index the entry-point properties so lookups start fast.
  • Spanner Graph if already on Spanner: querying the graph in the same database as the operational data avoids a separate system and keeps entities and relationships consistent.

How it maps to this tool

The knowledge graph is its own data source in the designer. Selecting it signals that the system needs relationship traversal, entity resolution, or lineage rather than similarity search alone.

It complements, it does not replace. The tool treats the knowledge graph as structured grounding in both purposes, sitting alongside vector retrieval. Choose the graph for connections and multi-hop reasoning, choose vector RAG for semantic similarity, and combine them when a question needs both.
