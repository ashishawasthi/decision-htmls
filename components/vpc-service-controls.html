Data-exfiltration perimeter

VPC Service Controls in agentic systems

VPC Service Controls draws a service perimeter around Google Cloud resources that blocks data movement to anything outside it, mitigating exfiltration and pinning data residency. The design rule that follows: under strict-PII or regulated requirements, put the sensitive services inside a perimeter so credentials alone cannot pull data out. It layers with IAM rather than replacing it: IAM is identity and access, VPC Service Controls is the perimeter. Reflects Google Cloud guidance as of May 2026.

Role

The role does not differ by purpose. For both interactive assistants and task automation, VPC Service Controls is the perimeter that keeps data in: it stops a stolen credential or misconfigured client from moving data to a project or network outside the boundary, and it pins where data can be accessed. It is an org and infrastructure control, not part of any single request path.

When to use, when not

Good fit

  • Strict-PII or regulated systems that must prevent exfiltration even when an identity is compromised.
  • Data-residency commitments that require sensitive data to stay within a defined set of projects and regions.
  • Designs that combine private connectivity (Private Service Connect) with a perimeter so traffic to Google APIs never traverses the public internet.
  • Multi-project estates where you want a hard boundary around the services that hold sensitive data, on top of IAM and Organization Policy.

Avoid

  • Low-sensitivity workloads where the operational cost of perimeter design and ingress and egress rules is not justified.
  • Treating it as access control: it does not grant or scope permissions. That is IAM, which you still need.
  • Treating it as field-level privacy: it does not redact PII inside the data. That is Cloud DLP.
  • Bolting it on late without planning ingress and egress rules: legitimate cross-project and partner flows can break, so design the boundary up front.

Choosing VPC Service Controls vs. alternatives

What you need Use instead Typical latency or cost
A boundary that blocks exfiltration and pins residency VPC Service Controls no direct charge; design and operations effort
Grant and scope who can do what IAM (layers with, does not replace) no direct charge; identity and access, not a perimeter
Org-wide guardrails on allowed configurations Organization Policy (complementary) no direct charge; constraints, not a data boundary
Private access to Google APIs without public IPs Private Service Connect (complementary) connectivity cost; private path, pairs with the perimeter
Remove sensitive data from content Cloud DLP (complementary) per unit processed; redaction, not a network boundary

Latency and cost profile

  • Latency: the perimeter is enforced at the control plane on API access, so it adds no meaningful latency to in-perimeter request paths. The real cost is design and operations: getting ingress and egress rules right.
  • Cost: no direct charge for VPC Service Controls itself. Budget instead for the engineering effort to design the perimeter and for any private connectivity (such as Private Service Connect) you pair with it.

Integration patterns and tuning

  • Wrap the sensitive services: place the projects and APIs that hold regulated data inside a perimeter so a valid credential cannot move that data to a project or network outside it.
  • Layer with IAM, do not swap: keep IAM for who can do what and add the perimeter for where data can go. The two together give least privilege plus a hard exfiltration boundary.
  • Pair with private connectivity and Organization Policy: use Private Service Connect so traffic to Google APIs stays private, and Organization Policy to constrain configurations, so the perimeter is one layer of a defense-in-depth posture.
  • Tuning: define ingress and egress rules for legitimate cross-project, partner, and CI flows, use access levels for trusted contexts, and test in dry-run mode before enforcing so you catch broken paths without an outage.

How it maps to this tool

The designer exposes this as the Data residency pin + VPC-SC override. It draws the exfiltration and residency boundary around the sensitive services in the architecture.

Strict-PII locks it on. When sensitivity is set to strict-PII, the rules engine forces the Data residency pin and VPC-SC on rather than leaving it optional, because residency and exfiltration control are table stakes at that tier. It sits alongside, not in place of, IAM (identity and access) and Cloud DLP (field-level redaction): the perimeter keeps data in, IAM controls who reaches it, and DLP removes sensitive values from the content itself.
