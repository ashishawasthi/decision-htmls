Spanner in agentic systems

Spanner is a strongly consistent, horizontally scalable relational database with synchronous multi-region replication. The one fact that drives the decision: it gives you SQL and multi-row transactions at scale without a single-primary bottleneck, including multi-region active-active. The design rule that follows: reach for Spanner when agent run state, sessions, or the transactional system of record must scale out or stay available across regions; otherwise a single-primary database is cheaper. Reflects Google Cloud guidance as of May 2026; verify against current Google Cloud docs.

Role

The role is the same across the tool's top-level purposes: Spanner is the durable transactional state store and system of record. It holds agent run state, conversation and session records, and the business rows the agent reads and writes transactionally. The reason to pick it over a cheaper database is scale and multi-region availability, not a different function.

When to use, when not

Good fit

  • Agent run state, sessions, and a transactional system of record that must scale horizontally beyond a single primary.
  • Multi-region active-active, where reads and writes are served close to users with strong consistency.
  • Workloads that need both relational SQL and strong external consistency at high write throughput.
  • High-availability requirements (regional or multi-region) backed by a strong SLA, with no manual sharding.

Avoid

  • Small, single-region workloads where a single primary is plenty. Cloud SQL is cheaper and simpler.
  • Postgres-compatible HTAP or in-engine vector retrieval. Use AlloyDB.
  • Massive key-value point lookups with no need for SQL or multi-row transactions. Use Bigtable.
  • Large analytical scans and aggregation. Use BigQuery, and keep it off the request path.

Choosing Spanner vs. alternatives

What you need Use instead Typical latency / cost
Single-primary relational at smaller scale, lower cost Cloud SQL low ms; per instance and storage
Postgres-compatible with HTAP and pgvector AlloyDB low ms; per vCPU and storage
Key-value at massive scale, no SQL or multi-row transactions Bigtable single-digit ms; per node and storage
Horizontal scale plus multi-region active-active relational Spanner low single-digit to low tens of ms; per node plus storage

Latency and cost profile

  • Latency: low single-digit to low tens of milliseconds for typical reads and writes, with multi-region writes paying a cross-region commit cost for strong consistency. It is built to hold these numbers as you scale out, not to beat a local single-node database on a single small query.
  • Cost: billed per compute capacity (nodes or processing units) plus storage, with multi-region configurations costing more than regional. Provisioned capacity means you pay for the footprint, not per query, so right-sizing nodes matters. Treat any figure as a rough range and verify against current Google Cloud pricing.

Integration patterns and tuning

  • Run state and sessions: store agent run records and session rows keyed so that related writes stay local. Design the primary key to spread load and avoid hotspots (avoid monotonically increasing keys).
  • Interleaving and locality: interleave child rows with their parent to keep transactional reads and writes within a split, reducing latency for the common access pattern.
  • Multi-region active-active: pick a multi-region configuration when you need writes served near users with strong consistency; accept the higher write latency and cost that buys.
  • Stale reads where safe: use bounded-staleness or exact-staleness reads for read-heavy paths that tolerate slightly old data, to cut latency and load versus strong reads.

How it maps to this tool

In the designer this is the State store input. The core choice is Spanner versus Cloud SQL: pick Spanner when the system needs horizontal scale or multi-region active-active, and Cloud SQL when a single primary in one region is enough.

Multi-region active-active is the tell. When the design calls for active-active writes across regions with strong consistency, the tool steers state to Spanner. If you only need a single-region primary, Cloud SQL keeps the cost lower for the same logical role.
