Streaming and event ingestion

Pub/Sub in agentic systems

Pub/Sub is global, managed messaging for event-driven ingestion and fresh, near-real-time context. The one fact that shapes the design: it decouples producers from consumers, so an event can trigger an automation run and stream fresh data into the system without anyone polling. The rule that follows: use it to kick off task automation on a ticket or event and to keep grounding data current, rather than as a request-path datastore. Reflects Google Cloud guidance as of May 2026.

Role by purpose

The right place for Pub/Sub flips with the top-level purpose you pick in the tool, because its job is to carry events and fresh data rather than answer a synchronous lookup.

Purpose Recommended role Fit
Interactive assistant Off-path freshness pipeline: stream fresh context into a store the assistant reads, not a source the request blocks on Conditional
Task automation Primary trigger (ticket or event) and ingestion backbone that starts and feeds runs Strong

When to use, when not

Good fit

  • Triggering automation: a ticket, alert, or domain event lands on a topic and starts an agent run.
  • Streaming fresh data into the system so grounding stays near-real-time instead of stale.
  • Decoupling and buffering: producers and consumers scale independently, and a backlog absorbs spikes without dropping work.
  • Fan-out: one event drives several consumers (the run, an audit log, an analytics sink) from a single publish.

Avoid

  • A synchronous request-path lookup. Pub/Sub is a message bus, not a key-value or query store.
  • Needing strict total ordering or exactly-once everywhere by default. It delivers at-least-once and ordering only within an ordering key.
  • Workloads that must stay portable across clouds or on-prem with no managed-service lock-in. Consider Kafka.
  • Simple low-volume polling where a scheduled pull is enough and a broker is overkill.

Choosing Pub/Sub vs. alternatives

What you need Use instead Typical latency / cost
Managed global event ingestion and triggers Pub/Sub sub-second delivery, per-volume cost
Portable, self-managed streaming with full control Apache Kafka (self-hosted or Managed Service for Kafka) low latency, you run and tune it
Routing events to services with filters and CloudEvents Eventarc (built on Pub/Sub) near-real-time, routing layer cost
Simplest possible integration, staleness tolerated Scheduled polling of the source poll interval staleness, low cost

Latency and cost profile

  • Latency: typically sub-second end to end for event delivery, which makes it a good trigger and freshness path. It is an asynchronous pipeline, so do not put it on a synchronous lookup that a user is waiting on.
  • Cost: priced mainly on message volume and throughput (bytes published, delivered, and any retained or replayed), so spend tracks event traffic. Rough ranges only as of May 2026; verify against current Google Cloud docs, since the figure is workload-dependent.

Integration patterns and tuning

  • Trigger pattern: a ticket or event publishes to a topic, a push subscription or Eventarc invokes the agent or a Cloud Run service, and the run begins. This is the core automation entry point.
  • Freshness pattern: stream changes onto a topic and land them in the store the agent grounds on, so context stays current without the request path reaching back to the source.
  • Ordering and delivery: delivery is at-least-once, so make consumers idempotent; enable ordering keys when per-entity order matters, and use dead-letter topics plus retry policies for poison messages.
  • Push vs pull: push (or Eventarc) suits serverless triggers; pull with flow control suits high-throughput workers that need to pace themselves.

How it maps to this tool

Pub/Sub is the Streaming data source in the designer and the automation Trigger (ticket or event) that starts a run. Selecting it tells the tool that work is event-driven and that grounding data can be kept fresh.

It moves the freshness, not the lookup. On the automation side Pub/Sub is a strong fit as the trigger and ingestion backbone. On the assistant side treat it as the off-path pipeline that keeps context near-real-time: stream fresh data into a store the request reads, rather than blocking the request on the bus itself.
