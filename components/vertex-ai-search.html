Agent Search in agentic systems

Agent Search (part of Agent Platform) is a managed retrieval engine. It ingests documents and crawls company websites into managed data stores, then serves blended semantic plus keyword search with built-in reranking and optional grounded answers with citations. It replaces the do-it-yourself stack of parse, chunk, embed, index, and rank with one service. The design rule that follows: make it the default whenever RAG is required, because the data has to be indexed in every RAG system anyway, and drop to a self-built Vector Search pipeline only when you need control the managed service does not expose. Reflects Google Cloud guidance as of May 2026; verify against current Google Cloud docs.

When to use, when not

Good fit

  • You need RAG and want documents and company website pages indexed without assembling a parse, chunk, embed, and index pipeline yourself.
  • Fast time to value: connectors for Cloud Storage, BigQuery, and websites (by URL pattern), with managed parsing, chunking, embedding, and a managed index.
  • Blended semantic plus keyword retrieval with built-in reranking and optional grounded answers with citations, out of the box.
  • Crawling and indexing company website or OSINT pages by URL pattern (basic or advanced website indexing).

Avoid

  • You need a custom chunker, a specific embedding model, or fine ANN tuning (HNSW M and efSearch). Use Vector Search with your own pipeline.
  • You run on-prem or self-host the stack. Managed Agent Search is a Google Cloud service.
  • You already operate a vector database (AlloyDB pgvector, Elasticsearch) and want to keep retrieval there.
  • Purely transactional or analytical lookups with no semantic retrieval. Use the right database instead.

Choosing Agent Search vs. alternatives

What you need Use instead
Full control of chunking, embeddings, and ANN tuning Vector Search + Document AI pipeline
Postgres plus vector in one engine, SQL pre-filtering AlloyDB (pgvector)
Hybrid lexical plus vector with self-hosting Elasticsearch
Managed end-to-end RAG over documents and websites Agent Search

Integration patterns and tuning

  • Data stores: create an unstructured data store for documents (Cloud Storage or BigQuery) and a website data store for site pages by URL pattern. Advanced website indexing crawls and refreshes pages on a schedule.
  • Rich-context parsing: the advanced parser is the Document AI Layout Parser, which chunks on structural boundaries (sections, tables, lists). For filterable grounding, import documents with structData metadata so extracted entities travel with each chunk and can pre-filter results.
  • Blended search and rerank: the engine fuses semantic and keyword signals and reranks, so you do not wire BM25 and a separate reranker yourself.
  • Grounded answers: use the answer and grounding APIs to return cited summaries, or return passages and let your own model generate the response.
  • Freshness: set a re-crawl and re-index cadence that matches how fast the source documents and pages change. The index is only as fresh as the last run.

How it maps to this tool

In the designer this is the Indexing and retrieval engine input, set to Managed: Agent Search by default whenever retrieval is on. The architecture diagram shows it as the store that retrieval reads, fed by a dashed offline branch from the index sources (documents and website). Switching the engine to Self-built reveals the Vector DB, HNSW, and ingestion-separation knobs and swaps the bill of materials to Document AI, Dataflow, a vector database, and BM25.

Default for RAG. Because every RAG system must index its data first, the tool defaults to managed Agent Search when retrieval is on, and treats the self-built Vector Search pipeline as the opt-in alternative for teams that need the extra control.
