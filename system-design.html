Vector search is a managed ScaNN service here. Both vector store options (Vertex AI Vector Search, the AlloyDB ScaNN index) and managed Agent Search run Google's ScaNN approximate-nearest-neighbour index, so there are no HNSW tuning knobs in this tool: recall and latency tuning is the service's job, and the design decisions that remain are the engine (managed vs self-built pipeline) and the store.

Workload and derivation assumptions. Every right-panel number derives from the inputs with these constants:

  • Requests / month = actors x actions per actor per day x 30.4 days.
  • Average QPS spreads the daily actions across the active window (active hours/week x 3600 / 7); peak applies the burst multiplier on top.
  • Vector index = corpus docs x 5 chunks/doc x 768-dim x 4 bytes, then sharded.
  • Latency p95 sums the critical-path line items; parallel grounding takes the slowest source, not the sum.
  • Model-call fan-out = agents x a damped ReAct loop (1 + (iterations - 1) x 0.5).
  • Egress bytes = output tokens x 4 bytes. Internet egress appears only for external or public audiences on GCP; hybrid designs bill a Partner Interconnect VLAN instead (private-only ingress).
  • Embedding refresh re-embeds 0.2% of the corpus per day at 5 chunks x 500 tokens per document; the one-time backfill is shown in the row detail but kept out of the monthly total.
  • Logging assumes ~12 logged bytes per token with a 36-month WORM hold; Cloud Trace assumes ~30 spans per request; DLP inspects prompt + response bytes; Model Armor screens ~2x the billed tokens.
  • Below the line: ops labor at illustrative loaded-FTE fractions (0.5 for GKE ops, 1 per 8 GPU nodes with a 0.5 floor, 0.5 for self-managed LangGraph) x $18k/mo, plus Enhanced support (the greater of $100 or a tiered 10/7/5/3% of spend). Neither enters the Total run-rate or Cost / 1k requests.
