Supported-peak sizing is split into two factors. Active volume hours / week: how many of the week's 168 hours your service actually carries traffic. 42 hr/wk (one fourth) is typical for daytime-only B2B usage; lower means the same monthly volume gets compressed into a narrower window. Average → supported peak: within those active hours, the burst multiple over the active-window average (1.5-2× for steady SaaS, 3-4× for typical user-facing, 5×+ for spiky consumer). Combined peak multiplier = (168 / active_hrs) × burst. Defaults (168/42) × 3 = 12×. When auto-linked, peak TPS = monthly volume × combined ÷ (730·3600 sec). WACC: capital tied up in on-prem hardware can't earn returns elsewhere. At 10% WACC, $290k of CapEx per server costs $29k/yr of foregone return. GCP CUD prices already bake financing into the hourly rate; on-prem doesn't. Utilization target: real on-prem clusters target 70-80% steady (not 100%) for ops headroom, model upgrades, traffic spikes. Sizing on-prem to 100% peak gives it an artificial advantage vs GCP self-host which is also sized to a utilization target. Set both knobs to the same value for a fair comparison.
Disaster Recovery (Active / Passive)
Enterprise SLAs typically require geographic redundancy. Warm standby keeps a half-capacity replica ready (failover in minutes, degraded service during outage). Hot standby is full duplicate (zero failover lag). DR doubles facility prep, electricity, and hardware support. Personnel can often be shared (we don't double that line).

What we model vs. what we don't: Token-based API costs (input + output + cache + grounding), VM hourly rates × 730 hours/month, 3-year amortized on-prem TCO with DR multiplier, capital opportunity cost (WACC), and a steady-utilization target on both self-host and on-prem. Throughput estimates per server are illustrative. Real numbers depend on batch size, KV cache budget, prefill vs decode mix, and serving framework (vLLM, TensorRT-LLM, SGLang). Per-node throughput assumes about 8,000 tok/s for a 70B-class model at FP8 on 8x H100, scaled by 70 / active-params, ×1.0 / ×2.0 / ×2.8 for BF16 / FP8 / INT4, with a 0.5× penalty for MoE routing. These constants dominate self-host and on-prem fleet sizing, so verify them against the companion serving-performance calculator and your serving framework.

Often-overlooked on-prem costs not modeled here (rough ranges per cluster, monthly, for a 3-server HGX H100 fleet):

  • Networking hardware: 400GbE NICs + spine/leaf switches, often InfiniBand fabric for multi-node serving. ~$50k-200k one-time CapEx amortized over 3 years ≈ $1.4k-5.5k / mo.
  • Internet bandwidth + peering / Direct Connect: $1k-3k / mo for production scale.
  • Storage: NVMe + NFS for model weights, KV cache spill, observability logs. $1k-2k / mo.
  • Software & licensing: monitoring stack (Datadog/Prometheus/Grafana), k8s/Slurm, optional NVIDIA AI Enterprise / vendor support. $2k-5k / mo at production scale.
  • Insurance + compliance: hardware insurance plus SOC 2 / ISO 27001 / HIPAA audits if you're enterprise. $500-2k / mo.
  • Backup power, UPS, generator: $50k-200k one-time + quarterly testing.
  • Procurement lead time: 5-6 months from order to production. If sized for a 24-month horizon, that's ~25% of useful life lost.
  • Cooling retrofit for high-density H100: 8x H100 server pulls ~5.6 kW IT; standard air cooling tops out around 15-20 kW/rack. Liquid cooling adds $50k-150k one-time per pod.
  • Residual value (credit, not cost): H100s typically retain 30-50% after 3 years. Not modeled; conservatively over-counts CapEx amortization.

Often-overlooked managed-API / self-host costs: data egress out of GCP ($0.08-$0.12/GB), Cloud Logging / Monitoring fees, regional NAT/PSC charges, longer-term tax / FX exposure on multi-year CUDs.

Rough rule of thumb: adding the items above can close the on-prem-vs-GCP gap by $10k-20k / mo at this fleet size. With realistic hidden costs included, GCP self-host on H100 CUD often lands within ~10% of fully-loaded on-prem TCO at sub-10-server scale.
