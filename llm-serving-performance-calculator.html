LLM Serving Performance Calculator

Roofline back-of-napkin for LLM inference: time to first token, per-token latency, throughput against batch size, KV-cache-bounded concurrency, and how many GPUs or Cloud TPUs an SLO needs. Deterministic, no LLM. This is a performance and capacity tool, not a memory-fit tool (for "does it fit in VRAM" use the memory calculator or Google's Gemma memory page).

Source & assumptions

  • Decode is memory-bandwidth bound: per step the weights (and active KV) are read from HBM once for the whole batch, so step time = (weight bytes + batch x KV bytes) / aggregate HBM bandwidth, and per-token compute = 2 x params x batch FLOPs is compared against it (the larger wins).
  • Prefill / TTFT is compute bound: 2 x params x prompt tokens FLOPs / (aggregate peak FLOPs x MFU).
  • KV bytes per token = 2 x layers x kv_heads x head_dim x bytes_per_elem (grouped-query aware; uses kv_heads, not query heads).
  • Accelerator FLOPs are dense BF16. NVIDIA's headline 1979 TFLOPS for H100 / H200 is the with-sparsity number; dense is about 990. Confirm against NVIDIA and Google Cloud TPU docs.
  • TPU FLOPs and bandwidth are per chip and approximate; treat them as order-of-magnitude for an interview, not a procurement spec.
  • This ignores speculative decoding, chunked prefill, prefix-cache reuse, and PagedAttention block waste (under about 4%). It is a first-principles floor, not a vLLM benchmark.
